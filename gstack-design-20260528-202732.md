## Problem Statement

Build a **purely static** webpage that simulates a high-fidelity Manufacturing Execution System for semiconductor fabs, referencing the visual and functional vocabulary of Tibco BusinessWorks, Siemens Opcenter Execution Semiconductor, and Applied Materials E3/FAB300. The demo must:

- Run entirely client-side (no backend, no auth, no server state)
- Present **multiple main MES functions** and master data, each with believable industry depth
- Continuously animate dynamic data — multiple categories of lot/material events streaming nonstop
- Use hash-based pseudo-random combinations to generate plenty of synthetic data
- Provide ~3 minutes of non-repeating dynamic content, then loop the timeline cleanly

The output is a hackathon deliverable but the audience targets **fab insiders + procurement decision-makers**, not general hackathon judges. "High fidelity" means an industry professional cannot dismiss it as a Dashboard mock.

## What Makes This Cool

The "whoa" is **cross-module event correlation under continuous random pressure**. A single scripted beat — a chamber drift triggering an SPC alarm — propagates visibly: the Equipment view flashes red, the Alarm Desk gains a row, the Fab Floor 2D hero strobes the bay, the KPI tile dips, and 30 seconds later a Maintenance event closes the loop. Industry insiders recognize the MES backbone behavior; procurement sees a product that "feels alive." Two hundred random background events fire underneath without distracting from the spine. It looks like a real Tier 1 MES because it behaves like one — within a 600KB static bundle.

## Constraints

- **Pure static deployment** — output is a `dist/` folder from `vite build`. No backend, no API, no DB, no auth. Deployable to GitHub Pages, S3, or zip+email.
- **Time-box**: ~3 days human / ~12 hr Claude Code (hackathon scale).
- **Runtime budget**: 3-minute loop. Scripted spine + hash-random background; loop boundary must be visually clean.
- **No real fab data, no real customer references, no real SECS/GEM transport.** S2F41/S6F11-style messages are mocked text. Hash combination of templates only.
- **Browser support**: modern Chromium/Edge/Firefox/Safari. No legacy.
- **Bundle target**: ≤ 800KB gzip for the dist payload (ceiling). Revised accounting after engineering review:
  - React base + react-dom ~140KB
  - Recharts ~95KB (Tremor dropped — confirmed in /plan-eng-review)
  - Framer Motion ~50KB
  - Radix primitives (shadcn used components) ~70KB
  - RxJS ~40KB (added per /plan-eng-review D1 — event bus replacement)
  - GSAP ~30KB (added per /plan-eng-review D4 — SVG floor animation)
  - @tanstack/react-virtual ~10KB
  - zustand ~3KB (retained for UI state — active nav, modals, selected entity)
  - clsx + tailwind-merge + date-fns ~15KB
  - **Vendor subtotal: ~453KB**. App code budget: ~250-350KB. Tighter than the original estimate but still feasible. If exceeded, cut Radix primitives via per-component shadcn install (drop Dialog/Sheet/Popover if unused).

## Premises

The following premises were explicitly confirmed by the user during office hours and shape every architectural decision below. If any premise turns out to be wrong, this design needs revision.

- **P1 — Deliverable shape**: Vite + React + TypeScript app, static `dist/` output, no backend.
- **P2 — Audience priority**: Fab insider (operators / MES PM / Fab IT) first, procurement/exec second. Hackathon judges are not the primary alignment target.
- **P3 — Hero scene**: 2D Bay/Chase fab floor + right-side live event stream as the landing view. Other modules exist behind a sidebar nav.
- **P4 — Data engine**: Scripted timeline JSON with 6 dramatic beats + 3 framing cues (shift boundaries, loop restart) + hash/PRNG random background events. 3-min loop with regenerated background.
- **P5 — Visual style**: Siemens Opcenter-leaning (blue-gray, dense data tables, hierarchical equipment trees). Applied Materials (dark SCADA) and Tibco (event-stream visualization) influences as accents, not the spine.
- **P6 — Mock realism**: SECS/GEM message *appearance* only (S2F41, S6F11, E10 state transitions, Lot ID format `LOT-YYWW#####`, SPC Western Electric rules). No real HSMS/TCP transport.
- **P7 — Master data**: Equipment, Recipe, Route, Operator, Product, Customer, Lot Genealogy — all synthetic, generated from templates × hash combination.
- **P8 — Distribution**: `dist/` deployed to GitHub Pages (primary) or zip handoff. No CI/CD pipeline design needed.

## Approaches Considered

### Approach A: Minimal Viable Demo (1 day human / ~4 hr CC)
Single Hero page. Side nav with 6 module names locked behind disabled state. 1 scripted beat. 80 random background events. Master data: 12 equipment / 25 lots / 5 recipes / 20 operators. Bundle ~250KB.
- Pros: lowest risk, ship-guaranteed within hackathon window; Hero concentrates attention.
- Cons: "main functions" requirement only half-satisfied; procurement clicking any sidebar item exposes the mock.

### Approach B: Full Module Tour (3 days human / ~12 hr CC) — **CHOSEN**
Hero + 6 functional sidebar modules (7 routes total): Production Tracking, Equipment Management, SPC/Quality, Recipe Management, Alarms & Events, KPI Dashboard. 6 scripted beats + 3 framing cues orchestrating cross-module correlation. Master data: 50 equipment / 200 lots / 30 recipes / 80 operators / 500+ wafer history records (rendered via virtualization in dense tables). Bundle ~750KB (ceiling 800KB — see updated accounting in Constraints).
- Pros: every sidebar entry survives a "click to test" probe; cross-module correlation is the wow moment for industry insiders; the user's plural "main functions" requirement is fully satisfied.
- Cons: highest effort; six modules each at 60% is harder than one at 90%.

### Approach C: Cinematic Auto-Tour (2 days human / ~8 hr CC)
Adds an auto-tour mode where the camera pans through 6 modules every 30 seconds with preset scripted animations per module. Manual override available. ~35% module depth per scene. Bundle ~450KB.
- Pros: "3-min non-repeating" requirement becomes the central design principle; kiosk-mode friendly for booth displays.
- Cons: industry-insider interaction ("let me click into lot X") is impeded by auto-tour; depth per module is shallow.

## Recommended Approach: B — Full Module Tour

**Why B and not A**: The user's literal phrasing "高仿真的主要功能**以及**主檔資料" calls out *plural* main functions. A satisfies one; B satisfies six. For the procurement audience layer, "click anywhere without it breaking" is the qualification test, and only B passes it.

**Why B and not C**: The fab-insider audience expects to drill into a specific lot, a specific tool, a specific recipe revision. Auto-tour fights that instinct. C is the right choice only if the deployment is unattended kiosk; for an interactive demo with judges or buyers leaning in, B wins.

**Data flow** (revised per /plan-eng-review + brainstorm review):
1. App boot → master data generated once from fixed seeds (deterministic master). `clock.ts` records `performance.now()` as t0.
2. **Pre-roll warmup** (added per brainstorm review): before starting the live clock, `timeline-engine` synthesizes ~30s of backdated history events using `PRNG(hash(0, -30..0))` and pushes them into the event bus as a batch. This ensures SPC charts have data points, KPI sparklines have trend lines, and the event stream has content on first paint. No module ever appears empty at load time.
3. `timeline-engine` reads `timeline.json` + spawns background event stream via PRNG seeded with `hash(loop_index, t)`. Engine ticks every 100ms reading `clock.elapsed()`.
4. Events published as RxJS `Subject<Event>` (replaces zustand store for event flow).
5. Each module subscribes to **topic-filtered observables** via typed selectors (e.g., `eventBus.ofTopic('lot.move')`, `eventBus.equipmentState(toolId)`) — no module subscribes to the full stream.
6. KPI module subscribes to `eventBus.ringBuffer(1000)` — a fixed-size ring buffer implemented via `scan()` (replaces `bufferCount(1000, 1)` to avoid per-event array allocation and GC pressure over long sessions). Recomputed values throttled to 500ms via `throttleTime` to limit Recharts re-renders.
7. **Sidebar badge counts**: `App.tsx` subscribes to lightweight derived observables (`eventBus.activeAlarmCount()`, `eventBus.activeLotCount()`, `eventBus.downEquipmentCount()`) and writes counts to `uiStore` via `throttleTime(1000)`. Sidebar renders badge pills next to Alarms, Production, and Equipment nav items. Other modules show no badge.
8. Every 180s the engine resets `loop_index++` and respawns background; spine replays identically. Pre-roll warmup re-runs with new `loop_index` seed to provide fresh backdated history. UI state (zustand `uiStore`) NOT reset — selected entity, active route, modals persist across loop boundary.

### Clock Semantics (`src/lib/clock.ts`)

```
                                t0 = performance.now() at boot
                                |
                                v
real time     ────────────────────────────────────────────────►
                                |<------- elapsed ------->|
                                                          |
                                                  clock.now() returns
                                                  elapsed in seconds

LOOP_DURATION = 180s
clock.loopT()        = clock.now() % 180          // 0..180 demo-time within loop
clock.loopIndex()    = floor(clock.now() / 180)   // monotonic loop counter

ON visibilitychange (hidden → visible):
  newElapsed     = performance.now() - t0
  expectedLoop   = floor(newElapsed / 180)
  if expectedLoop !== currentLoopIndex:
    // tab was hidden long enough that the loop advanced
    1. Cancel any in-flight timeline beats (they're stale)
    2. Set currentLoopIndex = expectedLoop
    3. Re-seed background PRNG with hash(expectedLoop, 0)
    4. Replay timeline spine from t=clock.loopT() onward
    5. Emit `shift.boundary` event with kind='loop-restart'
  else:
    // same loop, just catching up — timeline-engine's next tick will
    // fire any beats whose t has elapsed in one batch
```

This satisfies Success Criterion 6 (loop boundary cleanliness) for both the natural 180s boundary AND the tab-return case.

### Pre-roll Frame Spec (added per brainstorm review)

Resolves the reviewer concern about empty-on-first-paint. Before the live clock starts, the timeline engine generates synthetic history so every module has content on first render.

**Mechanism**:
1. At boot (after master data generation, before `clock.start()`), `timeline-engine.preRoll()` runs.
2. Pre-roll generates ~30s of backdated events using `PRNG(hash(loopIndex, t))` for `t = -30 .. -1` (1 event/sec average, same distribution as live background events).
3. These events are pushed into `eventBus` as a synchronous batch — subscribers receive them before the first React render commit.
4. The ring buffer populates with ~30 events, giving KPI `scan()` a warm start.
5. SPC chart renders with ~5 data points already plotted (not empty axes).
6. Event stream panel shows ~15 recent events (others scrolled off).
7. Sidebar badge counts are already populated on first paint.

**On loop restart** (t=180s or tab-return with loop advance): pre-roll re-runs with new `hash(newLoopIndex, -30..-1)` before spine replay begins, ensuring the same warm-start behavior.

**Bundle cost**: zero — reuses existing PRNG + event generation code paths.

### Error Boundary Strategy (added per brainstorm review)

Each module route in `App.tsx` is wrapped in `<ErrorBoundary fallback={<ModuleErrorState module={name} />}>`. Implementation:

- `ErrorBoundary.tsx` is a class component (React requirement) using `componentDidCatch`.
- On catch: logs error + stack to `console.error`, renders the module-specific error state from the Module Interaction States table (e.g., Equipment shows per-cell offline badge, SPC shows "Violation engine disconnected").
- A "Reload module" button calls `setState({ hasError: false })` to attempt re-mount — low cost, works for transient RxJS subscription errors.
- Errors do NOT propagate to other modules or crash the app shell / sidebar / top bar.

### Scripted Spine (timeline.json sketch)

6 dramatic beats + 3 framing cues = 9 timestamped events across the loop.

| t (s) | Kind | Beat | Affected Modules | Visible Effect |
|-------|------|------|---|---|
| 0:00 | framing | Shift start | Operators, Production | New shift roster, lots dispatched |
| 0:25 | **beat** | Lot priority insertion | Production, Equipment | Hot-lot inserted, routes reshuffle on Gantt |
| 0:50 | **beat** | Equipment minor alarm | Equipment, Alarms, Floor | Tool blinks amber, alarm row appears |
| 1:20 | **beat** | Chamber drift detected | SPC, Equipment | Western Electric Rule 2 fires, control chart violation |
| 1:45 | **beat** | SPC alarm → engineer ack | Alarms, SPC | Severity escalates red, operator ack flow |
| 2:10 | **beat** | Recipe revision pushed | Recipe, Equipment | Recipe diff appears, tool reloads recipe |
| 2:30 | **beat** | KPI recovery | KPI | OEE/Yield tiles climb back |
| 2:55 | framing | Shift handover prep | Operators | Loop boundary cue |
| 3:00 | framing | Loop restart (background re-seed) | All | Spine replays, background events new |

Background events (continuous, ~1/s average across the 6 modules):
- Lot moves between equipment (FROM, TO, lot_id)
- Recipe loads
- Operator login/logout
- Material consumption
- Wafer-level measurement points
- Minor (non-alarming) E10 state transitions
- Mock SECS messages on per-tool deep view when opened

## Success Criteria

The demo is considered done when **all** of the following hold:

1. `vite build` produces a `dist/` ≤ 800KB gzipped (target 700KB) that opens cleanly when served as static files.
2. Hero loads in < 1.5s on a clean Chrome session; first frame of animation visible within 200ms after that.
3. Each of the **7 routes** (Hero + 6 sidebar modules) is clickable from the sidebar and renders content distinct from the others, with at least one live-updating element.
4. The 3-minute scripted spine fires **all 6 dramatic beats** in correct order (+ 3 framing cues), and produces visible cross-module updates for at least 4 of the 6 beats.
5. Background events fire continuously without visible stutter — minimum 1 event/sec averaged, no module is silent for more than 5 seconds.
6. **Loop boundary cleanliness**: at t=180s the spine restarts and background re-seeds with: (a) no DOM unmount of the active route, (b) fade-through ≤ 300ms, (c) no flicker or state desync visible in any KPI tile or sparkline.
7. A senior MES PM looking at any module's master data for 30 seconds finds nothing implausible (Lot IDs, equipment names, recipe versions, E10 states, SECS message envelopes all pass sniff test).
8. A procurement-style click test (sidebar item → see real content) succeeds on all 7 routes.
9. **End-of-day smoke gate** (Day 1, 2, 3 close): every existing sidebar item opens without console error; spine plays end-to-end at least once before stopping work for the day.

### Typography

- **Sans**: IBM Plex Sans (Google Fonts, OFL) — `font-family: 'IBM Plex Sans', ui-sans-serif, system-ui`
- **Mono**: JetBrains Mono — `font-family: 'JetBrains Mono', ui-monospace, 'Cascadia Code', Consolas`
  - **Always use mono for**: Lot IDs, equipment IDs, SECS messages, timestamps, recipe versions
- Bundle impact: +70KB for IBM Plex Sans (acceptable trade for industrial vendor recognition over generic Inter)
- **Size scale**: xs=11px, sm=12px, base=13px, md=14px, lg=16px, xl=20px, 2xl=28px
- **Weight**: regular=400, medium=500, semibold=600
- **Line-height**: tight=1.2 (table rows), base=1.45 (body), loose=1.6 (rare)

### Spacing, Radius, Shadow, Motion

- **Spacing scale**: 0.5=2px, 1=4px, 2=8px, 3=12px, 4=16px, 5=20px, 6=24px, 8=32px
- **Radius**: 0=0px (data tables, equipment tiles — sharp industrial), sm=2px (buttons, chips), md=4px (KPI tiles, modals — limited use), full=9999px (status dots, avatars only)
- **Shadow**: thin=`0 1px 0 rgba(0,0,0,0.04)` (row hover), modal=`0 8px 24px rgba(0,0,0,0.12)`. NO decorative shadows.
- **Motion**:
  - duration: instant=80ms, quick=180ms, smooth=300ms (loop fade), deliberate=600ms (alarm escalation, KPI recovery)
  - easing: standard=`cubic-bezier(0.4,0,0.2,1)`, decel=`cubic-bezier(0,0,0.2,1)`

### AI Slop Guardrails (must be enforced in tailwind.config.js + component code)

1. **KPI tiles**: number-first layout (xl size, semibold weight) + sparkline below + 1px hairline `border-border`. **NO** icons-in-circles, **NO** gradient backgrounds, **NO** colored bars on top edge.
2. **Empty state copy**: terse industrial — `"No data — engine paused at t=<timestamp>"`. **NEVER**: emoji-laden friendly copy, "Oops!", "Looks like…", `🤖`.
3. **Buttons (shadcn override)**: `border-radius: 2px max`, `border: 1px solid border`, `box-shadow: none`. Hover = 5% darker fill, no elevation change.
4. **Recipe diff viewer**: monospaced, alternating row stripes `#FAFAFA`/`#FFFFFF`, additions = `#DCFCE7` background only (NO green text), removals = `#FEF2F2` background only (NO red text). No syntax highlighting outside diff markers.

**Event stream severity pinning** (added per brainstorm review): The `EventStream` component applies visual hierarchy:
