## Problem Statement
FabPulse is a high-fidelity semiconductor-fab demo with an MES layer and a full ERP layer (materials, business
partners, purchase orders, inventory, MRP, finance, document-flow cockpit) — all pure front-end, live on `master`.
The ask: extend it "outward" into an **SCM** layer emulating SAP/Oracle SCM (APO/IBP) master data + dynamic
simulation, as a richer demo. The defining constraint: ERP already owns *internal* procurement / inventory / MRP,
so SCM must extend the **cross-enterprise supply network** (suppliers, demand planning, in-transit logistics,
multi-site distribution, ATP, a control tower), NOT re-build what ERP has.

## What Makes This Cool
The hero is an **end-to-end supply-chain control tower**: a live network map — suppliers → FAB-01 → distribution
centers → customer regions — with shipments physically moving along lanes between nodes, ETAs ticking down, and
disruptions pulsing red. It closes the loop the ERP layer opened: an ERP purchase order becomes an **inbound
shipment in transit** that arrives and posts the goods receipt; a finished lot becomes an **outbound shipment**
that crosses the network and delivers to a customer, closing the sales order. You watch goods move across a
supply chain, not just documents change state. Three engines (MES, ERP, SCM) feeding one canvas.

## Constraints
- **Pure front-end, no backend.** Reuse the proven architecture: RxJS `eventBus` (already widened to `AppEvent`),
  seeded PRNG, shared `clock`, the `timeline-engine`/`bridge` patterns, zustand stores, the locked dark theme +
  primitives, `MasterDataModule<T>`, and the already-installed `React.lazy` route code-split.
- **SCM extends, never duplicates ERP.** No second procurement or inventory module.
- **Demo scale, performance first.** A handful of network nodes, dozens of in-flight shipments, ~1 event/s,
  lazy-loaded routes.
- **Desktop-only (≥1280px)** gate stays. Single locked dark theme.

## Premises (confirmed in session)
1. Pure front-end, extend the engine pattern — SCM data generated deterministically from existing MES+ERP data; a
   new SCM sim engine + `ScmEvent` folded into the `AppEvent` bus; reuse the bridge pattern.
2. SCM extends, doesn't duplicate ERP — inbound shipments reference ERP purchase orders; network inventory extends
   single-site inventory; demand plan feeds MRP; outbound/ATP reference sales orders; suppliers/customers are
   network nodes. No second procurement/inventory.
3. The logistics leg is the new dynamic spine — inbound: PO → shipment → in-transit (deterministic progress) →
   arrival → triggers the ERP goods receipt; outbound: `lot.complete` → shipment → in-transit → delivered → closes
   the sales order. A new `useShipments` store drives it; reset on the loop boundary.
4. SAP/Oracle SCM feel = APO/IBP-style: demand-planning time-bucket grid, a supply-network/control-tower map
   (nodes + flows), shipment tracker (status + ETA), supplier scorecards — on the locked dark theme.
5. Demo scale: few network nodes, dozens of in-transit shipments, ~1 event/s, route code-split (already set up).
6. New SCM nav domain → the sidebar becomes tri-domain (MES / ERP / SCM), collapsible; table screens reuse
   `MasterDataModule`, the map / demand grid / shipment tracker are bespoke.

## Approaches Considered
### Approach A: SCM sim engine + module suite  ← SELECTED
Mirror the ERP build. New `src/data/scm/` generators + a `scm-timeline-engine` (distinct seed) + a shipment
lifecycle driver (mirrors `bridge.ts`) + a `useShipments` store; `ScmEvent` widens `AppEvent`. Modules: Demand
Planning, Supply Network Control Tower (hero), Shipments/In-Transit, Supplier Scorecards, ATP. Tri-domain sidebar.
Effort L. Risk Med (3-engine coordination + network-map viz). Reuses: eventBus/clock/PRNG, the bridge pattern,
`useBridgedLots` pattern, `DenseDataTable`/`MasterDataModule`/`Panel`/`MetricTile`/`Gauge`/`StatusDot`/`chartTheme`,
the dark theme, the lazy-route shell.

### Approach B: Control-tower-first (single hero)
The network map IS the app; demand/shipments/suppliers are drill-downs off map nodes. Effort M-L. Risk Med-High
(map viz). Less breadth. Folded into A as the hero screen.

### Approach C: Logistics wedge (minimal)
Just the shipment lifecycle (inbound PO→GR leg, outbound FG→delivery leg) + the control tower; defer demand
planning + supplier scorecards. Effort S-M. Risk Low. Starts narrow (user chose all four areas).

## Recommended Approach — A (control tower grafted in as the hero)

### Events — `src/lib/scmEvents.ts` + the existing widened bus
`AppEvent` already = `MesEvent | ErpEvent` (the bus is generic-free over `AppEvent`). Add `ScmEvent` and widen:
`events.ts` → `export type AppEvent = MesEvent | ErpEvent | ScmEvent`. `eventBus.ts` needs NO change (already over
`AppEvent`). **`EventStream.tsx` is a HARD compile gate, not a soft "gain":** both its `eventMessage` switch and its
`TOPIC_META` (typed `Record<AppTopic, …>`) are **exhaustive over `AppTopic`** — the moment `ScmEvent` widens the
union, `tsc -b` fails until **every** new `scm.*` topic is added to both. Treat that as a required P0 sub-task.

`ScmEvent` topics (each with `t`) — **9 discrete-transition topics only** (deliberately NO per-tick progress event;
see the driver note for why): `scm.forecast.updated`, `scm.shipment.created`, `scm.shipment.departed`,
`scm.shipment.arrived`, `scm.shipment.delivered`, `scm.atp.promised`, `scm.supplier.asn`, `scm.disruption.raised`,
`scm.disruption.cleared`. Shipment in-transit `progress` is advanced **only in the `useShipments` store** (mirrors
`useBridgedLots.advanceLot`, which has no per-step bus event) — publishing a progress event per shipment per tick
(dozens × ~1/s) would flood the 1000-slot ring buffer and the re-sorting `EventStream`. The bus carries only the
discrete lifecycle transitions above; the store is the source of truth for live position.

### Data domain — `src/data/scm/` (deterministic, keyed off MES+ERP)
- `types.ts`: NetworkNode (id, kind: 'supplier'|'fab'|'dc'|'customer', name, region, x, y), Lane (from, to,
  mode: 'air'|'sea'|'truck', transitDays), Forecast (materialNo, buckets: number[], actuals: number[]),
  Shipment (shipmentNo, direction: 'inbound'|'outbound', fromNode, toNode, lane, refDoc {poNo|salesOrderNo},
  materialNo, qty, status: 'created'|'in-transit'|'arrived'|'delivered', **departureT, transitSeconds** — NO
  stored `progress` field; see ARCH-1 below: position is computed in the animation layer, not stored per tick),
  SupplierScorecard (bpNo, onTimePct, qualityPct, avgLeadDays, openAsns), AtpPromise (salesOrderNo, materialNo,
  promisedDate, available).
- generators: network nodes (suppliers from ERP vendor BPs, fab = FAB-01 plant, 2-3 DCs, customer regions from
  customer BPs grouped by country), lanes between them, forecasts per FERT material (deterministic curve +
  actuals), supplier scorecards (from BPs), seed initial in-flight shipments. `generateScmData(masterData, erpData)`.

### Simulation — `src/data/scm/scm-timeline-engine.ts` (distinct seed, e.g. `seededRng(loopIndex, 2000)`)
Mirrors the ERP engine. preRoll warms a few in-transit shipments + forecasts. Emits ambient SCM activity (~1/s):
forecast updates, supplier ASNs, occasional disruptions (a lane delay) + clears. Plus an SCM spine (a demand spike,
a port delay disruption mid-loop). Distinct seed so it never correlates with MES (0) or ERP (1000).

### The shipment driver — `src/data/scm/shipment-driver.ts` (mirrors `bridge.ts`) — eng-review LOCKED
The new dynamic spine. Owns shipment lifecycle via a `useShipments` zustand store (mirrors `useBridgedLots`:
`shipments[]` + add / transition(status) / reset actions + a `MAX_SHIPMENTS` cap with recycle, like `MAX_BRIDGED=60`).

- **ARCH-1 — position is computed, NOT stored per tick (eng-review Issue 1).** The store holds only DISCRETE state
  (lane, `departureT`, `transitSeconds`, status, refDoc, materialNo, qty). A `setInterval(tick, 200)` reads
  `clock.loopT()` and only fires **status transitions** (created → in-transit → arrived/delivered) into the store —
  exactly as `bridge.advanceLot` fires only on discrete step boundaries, not continuously. The Control Tower computes
  each dot's position per animation frame as `clamp01((clock.loopT() − departureT) / transitSeconds)` (a pure util,
  unit-tested) via CSS/SVG transform or framer-motion path animation. This avoids a 5 Hz × dozens-of-shipments
  store-write firehose that would tank the map's frame rate (the store-side twin of the bus firehose we already killed).
- **ARCH-2 — seed initial in-transit shipments on start AND every boundary (eng-review Issue 2).** Because inbound
  shipments arrive from `erp.po.created` (~0.2/s) and outbound from `lot.complete` (first one ~15–30 s in), and
  because `clock.onLoopBoundary` resets the store, the driver `seedInitial(loopIndex)` loads N deterministic
  (`seededRng(loopIndex, 2000)`) "already in-transit" shipments into `useShipments` — called on `start()` and again
  inside `onBoundary()` after the reset — so the hero map is populated on first paint and never blanks at the wrap.
  **Spread departureT (design Fix 9):** `departureT = boundaryT − rng()*transitSeconds` so seeded dots scatter
  mid-lane (computed position spans 0..1) on first paint AND at every wrap — never bunched at the origin nodes.
- **Inbound:** subscribe to `erp.po.created` → create an inbound `Shipment` (supplier node → FAB-01), set lane +
  `departureT`/`transitSeconds`; on the arrival transition emit `scm.shipment.arrived` AND a **PO-keyed**
  `erp.goods.movement` (`{movementType:'GR', storageLoc:'RAW', materialNo, qty}` referencing the arrived PO — SCM owns
  inbound receipt timing). **GUARD (mirror `bridge.ts:69`):** if no supplier node / lane to FAB-01 resolves, skip the
  shipment — no throw.
- **Outbound:** subscribe to `lot.complete` → create an outbound `Shipment` (FAB-01 → a DC → customer node) for the
  finished material; on the delivery transition emit `scm.shipment.delivered` (+ `scm.atp.promised` reconciliation).
  The bridge already emits the FG goods-receipt / order-Completed / invoice up-path on `lot.complete`; SCM adds only
  the physical delivery leg. **GUARD:** no outbound lane / customer node → skip.
- **Loop boundary (CRITICAL):** `onBoundary()` clears in-flight + `useShipments.reset()` THEN re-runs
  `seedInitial(loopIndex)` (per ARCH-2), or shipments orphan/blank at the 180s wrap.
- **Subscribe-before-emit ordering (CRITICAL, matches `App.tsx:142`):** `scmDriver.start()` must subscribe **before**
  `erpEngine.start()` — alongside `bridge.start()` — or the first `erp.po.created` / `lot.complete` are dropped.
  `scmEngine.preRoll()` runs alongside the other preRolls.
- Deterministic + module-scoped id counter (no randomness/wall-clock).

### ERP↔SCM event ownership — eng-review LOCKED (no double-count; light ambient damp)
Verified against the real code: **inventory is a static seeded snapshot** (`generateInventory`), read directly by the
Inventory and MRP screens. `erp.goods.movement` mutates **no** numbers — its only live consumers are the Cockpit
GR-lane (a rolling count + chips) and the EventStream feed. So there is **no inventory double-count risk**; the only
overlap is **cosmetic**. The ERP **bridge's** GR is the *outbound* FG receipt (`storageLoc:'FG'`) on `lot.complete` —
unrelated to the inbound PO leg SCM owns, so it stays. The only overlap with SCM's inbound GR is the ERP **engine's
*ambient*** `erp.goods.movement` / `erp.po.received` noise (random, unkeyed, `erp-timeline-engine.ts:85-100`).
**Decision (ARCH-3, LOCKED):** SCM emits the PO-keyed inbound GR on arrival, AND we **lightly damp the ERP engine's
ambient GR rate** (narrow the `roll < 0.72` band so ambient GR is reduced, not zero) so the Cockpit GR-lane reads as
real SCM arrivals. This is a one-line edit to shipped code → it carries a **mandatory regression test** (see Test Plan).

### UI — modules + tri-domain nav
Sidebar becomes **tri-domain** (MES / ERP / SCM), collapsible groups (extend `DOMAINS` in `App.tsx` + the lazy
route map + `ModuleRoute`/`SelectedEntity`/`BadgeCounts` unions). New modules (reuse `MasterDataModule` for tables,
bespoke for the rest):
- **Supply Network Control Tower** (HERO, bespoke SVG) — network map: nodes (suppliers/fab/DCs/customers) at
  **fixed hand-placed `x,y` coordinates baked into the generator, in a fixed SVG `viewBox`** (no geo/map library, no
  computed layout — legible at demo scale), lanes drawn between them, shipments animate along lanes as glowing dots
  (the geographic analog of the wafer-flow particles) — **position computed from `clamp01((loopT−departureT)/transitSeconds)`
  in the animation layer (ARCH-1), not from a stored `progress`** — disruption alerts pulse red on affected lanes
  (`.animate-sonar`). Count/legend overlay. **Perf guidance:** animate dots via CSS/SVG `transform` (or transform-only
  framer-motion), cap concurrently animated dots, honor `prefers-reduced-motion`. Reuse the framer-motion + SVG
  approach from `BayLayout`/`Cockpit`. (A *distinct* hero from the ERP `Cockpit` swim-lanes — geographic network vs.
  document pipeline — so it must clearly out-class it, not read as a reskin.) **Full visual grammar — dot model,
  arrival beat, node tiers, lane/overflow rules, depth stack, KPI rail, legend, interaction — is specified in the
  Visual Design Spec (section A) below.**
- **Shipments / In-Transit** (`MasterDataModule` + status) — table of shipments: shipmentNo, direction, from→to,
  material, qty, status chip, ETA, an in-transit progress bar (computed from the same position util, not a stored
  field). Live via `useShipments`.
- **Demand Planning** (bespoke, IBP-style) — per-material forecast grid (heat-tinted variance cells) + a `ComposedChart`;
  see Visual Design Spec **C** for the full grammar. Feeds MRP narrative.
- **Supplier Scorecards** — a `panel panel-hover` **card grid with radial `Gauge`s** (NOT a flat `MasterDataModule`
  table; design Fix 6); on-time %, quality %, avg lead, open ASNs; drill-in. See Visual Design Spec **C**.
- **ATP / Order Promising — folded into Sales Orders, NOT a standalone module (eng-review Issue 4, DRY):** add a
  `promisedDate` + ATP-status column to the existing ERP Sales Orders screen, plus one compact ATP availability panel
  (reads inventory + inbound shipments + production). Avoids a third screen over the same orders/inventory. (Net-new
  module count drops 5 → 4.)

### Visual Design Spec — `/plan-design-review` LOCKED (apply all; 6/10 → target 9-10)
The review's core finding: the plan named the right primitives but **under-specified which carry the visual load**, so
a naive build would ship flat. Every spec below pins to an EXISTING primitive (no new colors, no new keyframes, one
net-new SVG hero). Cited line numbers are calibration targets in `BayLayout.tsx` / `Cockpit.tsx` / `index.css` /
`tokens.ts` / `chartTheme`.

**A. Control Tower — the hero (Fixes 1,2,3,5,10; missing-dim: interaction).**
- **Dot model (Fix 1, the most-seen object) — RESOLVE the ambiguity:** each shipment is a controlled SVG `<g>`
  positioned every `requestAnimationFrame` from `shipmentPosition(clock.loopT(), departureT, transitSeconds)` applied
  as `transform="translate(x,y)"` — mutated **imperatively via refs, never React setState**, so the lane/node layer
  never re-renders during transit (this is what actually guarantees 60fps; ARCH-1 only kills the store firehose).
  **NOT** a framer-motion auto-tween (it would fight `loopT` as the source of truth). **Reject `fp-flow`/offset-path
  for dots** — it runs on an independent CSS timeline that can't be slaved to `clock.loopT()`, so it would desync from
  ETAs and the 180s wrap (it may still be used for purely decorative ambient lane sparks).
- **Dot composition** = the FabFloor wafer-flow comet (`BayLayout.tsx:258-276`): blurred outer halo (r≈7, blur 3px) +
  white-hot `#CFFAFE` core through `filter="url(#fpTileGlow)"` (define the filter in the map `<defs>`) + a faint
  `strokeDasharray="3 4"` wake; tinted by lane mode (air=accent, sea=accent-2, truck=accent-3). +`0 0 6px` glow +1px
  `surface-2` stroke for separation from the grid/vignette.
- **Arrival/delivery WOW beat (Fix 2 — the payoff):** on the inbound arrival transition fire a one-shot
  success-tinted `.animate-sonar` ring + a brief `.animate-pulse-glow` node flash at FAB-01, time-locked to the
  PO-keyed `erp.goods.movement` GR; on outbound delivery fire a sonar pop at the customer node as the SO closes; the
  dot fades `opacity→0` over ~300ms. These are the geographic echo of the Cockpit freshest-chip glow — the loop reads
  without explanation. On departure the dot enters with `fp-rise` (240ms) at the origin node.
- **Node tiers + identity (Fix 3) — NOT four identical dots:** each kind a distinct glyph + tone + tier + lucide icon
  matching the sidebar, as a labeled cluster (mono name + `ink-3` region sub-label + count badge), each with a
  `fpTileGlow` bloom halo: supplier=`accent-3` (upstream, smaller ring), **FAB-01 = `accent` #22D3EE, largest/brightest
  anchor, `.glow-cyan` + `.hud-frame` brackets**, dc=`accent-2` mid hub, customer=`success` #34D399 endpoint.
- **Lanes + overflow guard (Fix 5 — the FabFloor overflow lesson, in map terms):** lanes are curved SVG `<path>` arcs
  (quadratic, slight bow so crossings separate), routed **node-EDGE to node-edge** (offset by node radius so a dot at
  clamp 0/1 never occludes a node), per-mode `strokeDasharray` marching-ants. The map is a fixed `viewBox="0 0 1000 560"`
  SVG with `className="w-full h-full"` inside a `Panel hud-frame relative flex-1 min-h-0 overflow-hidden` wrapper
  (mirror `FabFloor/index.tsx:27` + `BayLayout.tsx:104`) so it **intrinsically clips — cannot overflow the panel**.
  All node `x,y` on a documented **≥24px safe inset**; each `NetworkNode` carries `labelSide(left|right|top|bottom)` →
  SVG `<text>` `textAnchor` per side, `pointer-events-none`, in a dedicated label band (no collisions). **Node budget:**
  ≤5 suppliers, FAB-01, 2-3 DCs, ≤4 customer regions; excess vendors/customers aggregate into an "Others/region" node
  (finite authored coords, like BayLayout's 8 fixed bays). Scale by viewBox uniform-scaling only; verify at 1280px.
- **Depth + KPI rail (Fix 10):** carry BayLayout's depth stack — `fpFloorGrid` `<pattern>` wash behind the network,
  `fpTileGlow` bloom on nodes, `.scan-sweep` + `.hud-frame` on the hero frame. Crown the panel with a 4-5 `MetricTile`
  rail (In-Transit / Late / On-Time% / Open Disruptions / ATP-at-risk) so a number is the first focal point.
- **Legend = HUD overlay:** `.glass` + `.hud-frame` bottom-left (BayLayout `glass rounded-md absolute bottom-2 left-2`):
  glowing swatches per node kind, lane-mode key, a `.animate-sonar` rose disruption chip, live counts in tabular-nums.
- **Interaction/drill-in (missing-dim):** hover a dot → `ChartTooltip`-style ETA/PO tooltip; click a node/lane → opens
  its shipment list (reuse `DrillInPanel`) and cross-highlights the Shipments table. A tower you can interrogate reads
  as a tool, not a poster.
- **Ambient liveliness + reduced-motion (Fix 11):** between events, lanes keep a slow marching-ants shimmer + nodes a
  gentle `.animate-pulse-soft` breath. Under `prefers-reduced-motion` (matchMedia early-return like `BayLayout:86-87`)
  skip the rAF glide + arrival glow — dots render at their static `shipmentPosition()` coord; CSS already no-ops
  `.animate-sonar`, so disruption MUST stay legible statically: a persistent `--sem-critical` lane stroke + a
  non-animated alert chip on the node (never animation-only).
