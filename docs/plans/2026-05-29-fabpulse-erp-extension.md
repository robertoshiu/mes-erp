## Problem Statement
FabPulse is a high-fidelity, purely-static semiconductor-fab **MES** demo. The ask: extend it
"upward" into an **ERP** layer that emulates SAP/Oracle-style master data and various dynamic data
simulation, as a richer demo — not a production system. The win condition is a screen-rich, *alive*
demo that visibly ties an ERP (orders, planning, inventory, finance) to the existing shop floor.

## What Makes This Cool
The hero is a **living ERP↔MES bridge**: a sales order arrives → MRP plans it → a production order is
released → it drops onto the existing Fab Floor as a real lot → the lot finishes on the floor → it flows
back up as a goods receipt, yield, and cost posting. You watch a business document turn into a wafer lot
and a wafer lot turn into money, in real time, on one dark command-center canvas. No other fab demo shows
the full ERP-to-fab thread animated end to end.

## Constraints
- **Pure front-end, no backend.** Reuse the proven architecture: RxJS `eventBus`, seeded `mulberry32`/
  `cyrb53` PRNG, `clock` (180s loop), `timeline-engine`, `zustand` UI store. ERP master data is generated
  deterministically; ERP dynamics come from a new simulation engine, not a server.
- **Demo scale, performance first.** Hundreds of master records (not millions), virtualized tables
  (`DenseDataTable`), ~1 event/s simulation cadence, single `dist/` bundle (keep gzip reasonable; lazy-load
  heavy modules if the bundle grows past ~300KB gzip).
- **Visual language is fixed.** Everything inherits the existing "Siemens Cyan" dark theme, tokens, and
  shared primitives. SAP "feel" = transaction-code-style screens + header/line-item forms + document flow,
  NOT a pixel clone of SAP GUI/Fiori.
- **Desktop-only (≥1280px)** gate stays.

## Premises (confirmed in session)
1. Pure front-end, no backend — extend `eventBus` + PRNG + `timeline-engine`; ERP master = deterministic
   generators; ERP dynamics = a new ERP sim engine + new event topics on the same clock.
2. The **ERP↔MES bridge is the spine**: SalesOrder → MRP → ProductionOrder flows *down* into a MES lot;
   MES completion flows *up* into goods receipt / yield / cost. Shared keys: existing lot/product/equipment.
3. Master data **extends, not replaces**: `products`→Material Master, `customers`→Customer/BP,
   `equipment`→Work Centers; then add Material/Vendor/BOM/Cost-Center/Plant, all cross-referenced.
4. SAP/Oracle quality = transaction-code screens + header/detail master forms + document flow graph, on the
   existing dark theme.
5. Demo-scale, performance-first: hundreds of records, virtualized tables, ~1 event/s.

## Approaches Considered
### Approach A: ERP simulation engine + module suite  ← SELECTED
Mirror the existing MES structure. New `src/data/erp/` generators + a new `erp-timeline-engine` emitting ERP
events on the same `clock`; a suite of ERP modules; a Document-Flow Cockpit hero screen (idea borrowed from
Approach B); a cross-engine bridge between ERP and MES. Effort L. Risk: Medium (two-engine event coordination).
Reuses: eventBus, PRNG, clock, timeline-engine pattern, DenseDataTable, Panel/MetricTile/Gauge/StatusDot,
chartTheme, DrillInPanel, design system.

### Approach B: Document-flow control tower (single hero)
One SAP-style document-flow network graph as the centerpiece + master drill-downs. Effort M-L. Risk: Med-High
(graph viz). Elegant narrative but thinner master-data breadth. Folded into A as the "Document Flow Cockpit".

### Approach C: O2C vertical wedge first, breadth later
Ship Order→MRP→ProductionOrder→Lot→completion→GoodsReceipt end-to-end first, breadth in later passes. Effort
S-M. Risk: Low. Fastest to a living thread, but starts narrow (user chose all four ERP areas).

## Recommended Approach — A (with B's cockpit grafted in)

### Data domain — `src/data/erp/` (deterministic generators, seeded)
Mirror `src/data/master/`. Each generator keys off existing master data so everything cross-references.

- `materials.ts` — **Material Master** (SAP MARA-flavored). Derived from `products` (finished goods) PLUS
  generated raw materials (silicon wafers, photoresist, target metals, process gases) and semi-finished.
  Fields: materialNo, type (FERT/HALB/ROH), description, baseUoM, materialGroup, plant, valuationClass,
  standardCost, leadTimeDays. ~120-180 rows.
- `businessPartners.ts` — **Customer/Vendor Master (BP)**. Customers extend existing `customers`; add vendors
  (ASML, LAM, AMAT, TEL, gas/chemical suppliers — reuse the vendor names already in `equipment.ts`). Fields:
  bpNo, role (customer|vendor|both), name, country, paymentTerms, incoterms, creditLimit.
- `bom.ts` — **Bill of Materials**. For each finished-good material, a BOM of raw/semi materials + quantities,
  loosely consistent with the product's `route` step types. Header + components (recursive one level).
- `workCenters.ts` — **Work Centers** mapped from existing `equipment` bays/tool types (capacity, cost center,
  available hours). This is the explicit MES↔ERP master link.
- `costCenters.ts` + `glAccounts.ts` — **Cost Centers** (per bay/area) + a small **GL** chart of accounts
  (inventory, WIP, COGS, revenue, variance) for postings.
- `plants.ts` / storage locations — **Org structure** (FAB-01 plant, storage locations: raw, WIP, FG).
- `salesOrders.ts` — **Sales Orders** (header: customer, order date, requested date; lines: material, qty,
  net price). Seed an initial backlog; more arrive via the sim engine.
- `purchaseOrders.ts` — **Purchase Orders** to vendors for raw materials (header/line, delivery date).
- `productionOrders.ts` — **Production Orders** linking a finished material + BOM + routing + target qty;
  status network (Created→Released→InProcess→Completed). Released orders are what bridge into MES lots.
- `inventory.ts` — **Stock** per material × storage location, with on-hand/committed/available.
- `erp/index.ts` — `generateErpData(masterData)` returns the cross-referenced ERP dataset (so MES master is
  the single source of truth for shared entities).

### Events — `src/lib/erpEvents.ts` + one widened bus
Add `ErpEvent` with topics: `erp.order.created`, `erp.mrp.run`, `erp.plannedorder.created`,
`erp.prodorder.released`, `erp.prodorder.status`, `erp.goods.movement` (GR/GI + movementType),
`erp.po.created`, `erp.po.received`, `erp.gl.posting`, `erp.invoice.created`.

**Decision (resolves the earlier open question): ONE bus, widened union — NOT a second bus.**
`createEventBus`/`EventBus` in `src/lib/eventBus.ts` are hardcoded to `MesEvent` (not generic), so
`createEventBus<ErpEvent>()` will not compile and generic-izing it would touch every module's `EventBus`
prop. Instead, in `events.ts` add `export type AppEvent = MesEvent | ErpEvent` and switch `eventBus.ts`
from `MesEvent`→`AppEvent` (~3 lines: internal `Subject`, ring buffer, `all$`, `ofTopic<T extends
AppEvent['topic']>`). Modules calling `eventBus.ofTopic('equip.state')` keep working (the topic narrows to
the right variant — no change). The only prop to widen is `EventStream`'s `events$: Observable<MesEvent>` →
`Observable<AppEvent>` (it consumes `all$()`); add ERP entries to its `TOPIC_META`/`eventMessage`. Lowest-churn
path, and it honors Premise 1 ("extend the eventBus").

**Also add a MES `lot.complete` topic** to `MesEvent` — there is no completion signal today (see Bridge).

### Simulation — `src/data/erp/erp-timeline-engine.ts`
Same shape as `timeline-engine.ts`: `preRoll()` warms an in-flight set of orders so no ERP screen is empty on
first paint; `start()` runs on the shared `clock` emitting ~1 ERP event/s plus an **ERP spine** of scripted
beats (a hot rush order; an MRP run that creates planned orders; a PO delayed → raw-material shortage; a
month-end GL close). Deterministic (seeded PRNG, no `Math.random`/`Date.now` in render).

### The bridge — `src/data/erp/bridge.ts`
The spine. **The bridge OWNS the lifecycle of bridged lots.** The existing engine's background `lot.move`
carries a *random* `routeStep` (`timeline-engine.ts:41`) and never deterministically completes, and there is
no completion event today (`events.ts` has only `lot.move`). So the bridge cannot passively "observe
completion" — it drives it.

- **Shared registry:** a **zustand store `useBridgedLots`** (mirrors `uiStore` — reactive single source of
  truth; a bare module array would not trigger React re-renders). BOTH the MES Fab Floor / Production table and
  the ERP Production Orders module read it via the hook; the bridge writes via store actions. The seeded
  `masterData.lots` set stays read-only — the Production table renders `masterData.lots ∪ store lots` — so a
  bridged lot is literally one object shown in three places. [locked: eng-review A1]
- **Down:** on `erp.prodorder.released`, the bridge creates a `bridgedLot` (product/route/qty from the order)
  and schedules a **deterministic** step progression — emitting `lot.move` for steps 0..N over the order's
  lead time on the shared `clock` — then emits a new `lot.complete` at the final step.
- **Up:** the up-path triggers on `lot.complete` (NOT on random background moves): emit `erp.goods.movement`
  (GR into FG stock), set the production order Completed, post `erp.gl.posting` (WIP→FG→COGS) + a yield number,
  and close the originating sales-order line.
- **Timing:** the bridge subscribes AFTER both engines' `preRoll()`, so pre-roll-released orders are warmed
  into `bridgedLots` once (not double-injected). Background `lot.move` stays purely cosmetic and is ignored.

### UI — new modules + nav reorg
Group the sidebar into **MES** (existing Operations/Quality/Command) and **ERP**. New routes/modules
(reusing `DenseDataTable`, `Panel/PanelHeader`, `DrillInPanel`, `MetricTile`, `Gauge`, `chartTheme`,
`StatusDot`):
- **Document Flow Cockpit** (hero, from Approach B) — a status-node graph SalesOrder→PlannedOrder→
  ProductionOrder→Lot→GoodsReceipt→Invoice with live highlighting as events fire; click a node to drill in.
  Reuse framer-motion + SVG (same toolkit as the wafer-flow particles).
- **Materials** — Material Master browser (transaction-code-style: `MM03`-flavored header tag), header/detail
  drill-in (basic data, valuation, BOM tab, stock tab).
- **Business Partners** — customers + vendors, role filter, credit/terms detail.
- **BOM Explorer** — expandable BOM tree per material (reuse the genealogy-link pattern from Production).
- **Sales Orders (O2C)** — order list + line items + status; "create rush order" demo affordance.
- **Planning / MRP** — a planning board: demand vs supply per material, an MRP-run beat that generates planned
  orders; coverage/shortage indicators (red when a PO is late). Charts via `chartTheme`.
- **Production Orders** — order list + status network; shows the link to its MES lot (click → jump to Fab Floor).
- **Inventory & Movements** — stock per material×location + a live goods-movement ledger (GR/GI feed, reuse
  `EventStream` styling).
- **Procurement (PO)** — purchase orders + vendor delivery simulation.
- **Finance** — cost centers, a live GL posting feed, simple WIP/inventory valuation + a P&L-ish KPI strip
  (reuse `MetricTile`/`Gauge`).

`uiStore`: extend the unions explicitly —
- `ModuleRoute` += `'erp-cockpit' | 'materials' | 'business-partners' | 'bom' | 'sales-orders' | 'mrp' |
  'production-orders' | 'inventory' | 'procurement' | 'finance'`.
- `selectedEntity.type` += `'material' | 'businessPartner' | 'salesOrder' | 'plannedOrder' | 'prodOrder' |
  'purchaseOrder' | 'costCenter' | 'glAccount'`.
- `BadgeCounts` += `openOrders` | `shortages` | `latePOs` (extend the badge subscription `useEffect` in `App.tsx`).

**Module DRY [locked: eng-review Q2]:** extract a generic `MasterDataModule<T>` (config: columns, drill-in
renderer, data, `selectedKey`) for the 7 table+drill-in screens (Materials, Business Partners, Sales Orders,
Production Orders, Inventory, Procurement, Cost Centers/GL). Keep BESPOKE modules for the 4 that aren't tables
(Document-Flow Cockpit, MRP planning board, Finance charts, BOM tree). Do not over-generalize into one engine.

**Code-split [locked: eng-review P3]:** convert `App.renderModule`'s `switch` to `React.lazy` + `<Suspense>`
(fallback = a `Panel` skeleton), so each module (MES + ERP) is its own chunk. Land this during the P1 nav reorg,
not as a P5 retrofit — recharts (MRP/Finance) and framer-motion (Cockpit) then load lazily.

### SAP/Oracle "feel" cues (cheap, high-impact)
- Transaction-code chips on headers (e.g. `VA03`, `MM03`, `CO03`, `MD04`) — pure flavor.
- Header + line-item master forms (the SAP two-pane shape).
- Document-flow graph + status networks (Created/Released/Completed) with colored nodes.
- Org breadcrumb (Client → Plant FAB-01 → Storage Loc).

## Open Questions
- ~~Bus shape~~ **RESOLVED:** one widened `AppEvent` bus (see Events). No second bus.
- Planned→Production release trigger: a scripted ERP spine beat releases planned orders (so the Cockpit shows
  a live PlannedOrder→ProductionOrder transition) plus a manual "release" affordance on the MRP board —
  rather than silent auto-release.
- Cockpit event→node mapping: every `erp.*`/`lot.*` event carries its document/lot id; the Cockpit keys nodes
  by document id and highlights the node/edge matching the latest event. Fixed swim-lane columns (not a force
  layout) for legibility at demo scale.
- Bundle: code-split ERP routes via dynamic `import()` starting in P1 (when the nav reorg lands), not as a P5
  retrofit; recharts (already a dep) on the MRP/Finance screens is the main weight to watch.
- Completed sales orders/POs: cap + recycle over the 180s loop, like the MES ring buffer.

## Success Criteria
- A first-time viewer can watch one sales order become a fab lot and come back as stock + a GL posting,
  without explanation, within one 180s loop.
- All four ERP areas (master data, O2C/production, inventory/procurement, finance) have a populated, live
  screen on first paint (pre-roll warms them).
- Zero console errors; `tsc -b` clean; `vite build` succeeds; 60fps-ish on the cockpit animation.
- Every new screen reuses the existing design system (no light/flat panels, no emoji icons).
